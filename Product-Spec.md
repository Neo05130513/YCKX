---

## 1. 产品愿景

建设一套以 BT 码锂电池组为核心的综合管理系统，统一管理 8 个第三方平台的电池数据，打通电池资产、租赁订单、客户账单、售后报修、财务管理和业务审批流程。

系统目标不是再做一个泛泛的“内部管理平台”，而是围绕电池租赁业务，把资产流转、客户履约、设备运行和财务回款串成一条可追踪、可查询、可审批、可对账的业务链路。

---

## 2. 系统定位

### 2.1 系统组成

| 系统端 | 使用对象 | 说明 |
|--------|----------|------|
| 内部管理后台 | 公司内部员工 | 设备、告警、租赁订单、固定资产、售后、财务、OA审批、系统配置 |
| 客户移动端网页 | 外部客户 | 查看订单、电池、账单、欠款、提交报修、确认收货 |

### 2.2 核心业务对象

| 对象 | 定义 |
|------|------|
| BT码 | 一组锂电池的唯一识别码 |
| 电池组 | 系统核心资产，计量单位为“组” |
| 客户 | 租赁业务主体 |
| 租赁订单 | 销售创建的客户租赁业务单据 |
| 合同 | 附属于租赁订单的合同信息或附件 |
| 出库单 | 资产部门基于订单创建的电池出库记录 |
| 月度账单 | 系统按订单账单日自动生成的应收账单 |
| 报修单 | 客户或内部人员提交的电池故障处理单 |

## 5. 功能需求

### 5.1 账号权限

#### 功能描述

系统需要支持内部员工账号和客户账号。客户账号由后台开通，一个客户公司可开多个账号。

#### 功能清单

| 功能模块 | 功能描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| 内部账号管理 | 创建、编辑、禁用内部员工账号 | P0 | 管理员可维护员工账号状态 |
| 客户账号管理 | 为客户公司创建多个登录账号 | P0 | 客户账号可登录客户移动端网页 |
| 角色权限 | 配置系统管理员、销售、资产、财务、技术、售后等角色 | P0 | 不同角色看到对应菜单和数据 |
| 客户权限 | 区分客户管理员和客户员工 | P0 | 客户员工不可查看账单金额和欠款金额 |
| 密码管理 | 支持后台重置客户或员工密码 | P1 | 用户忘记密码时可由后台处理 |

#### 客户权限规则

| 功能 | 客户管理员 | 客户员工 |
|------|------------|----------|
| 查看订单 | 是 | 是 |
| 查看电池 | 是 | 是 |
| 查看账单金额 | 是 | 否 |
| 查看欠款总额 | 是 | 否 |
| 查看已付金额 | 是 | 否 |
| 查看逾期金额 | 是 | 否 |
| 提交报修 | 是 | 是 |
| 查看报修进度 | 是 | 是 |
| 确认收货 | 是 | 是 |

### 5.2 设备管理

#### 功能描述

设备管理用于汇总 8 个第三方平台的电池运行数据。第三方平台提供接口，本系统主动拉取数据，并按 BT码统一清洗和展示。

#### 功能清单

| 功能模块 | 功能描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| 平台接口配置 | 配置 8 个平台接口地址、认证方式、同步频率 | P0 | 管理员可维护平台接口配置 |
| 定时数据拉取 | 系统主动按配置频率拉取平台数据 | P0 | 可查看同步成功或失败记录 |
| 字段映射 | 将不同平台字段映射为统一字段 | P0 | 同类字段可统一展示 |
| 数据清洗 | 统一 BT码、单位、状态、异常值 | P0 | 清洗后的数据可进入统一电池视图 |
| 电池列表 | 按 BT码展示全部电池运行数据 | P0 | 可查询 BT码对应电池 |
| 电池详情 | 展示单组电池的运行数据、来源平台、更新时间 | P0 | 可查看指定 BT码详情 |
| 同步日志 | 记录同步时间、结果、数量、异常原因 | P1 | 接口异常可追踪 |
| 异常数据识别 | 识别 BT码缺失、重复、字段异常 | P1 | 异常数据不应静默进入正常列表 |

#### 关键规则

- 第三方平台数据为设备运行数据来源。
- 本系统以 BT码作为统一电池资产主键。
- 设备运行状态和资产业务状态必须分开管理。
- 第三方平台数据暂按只读处理，是否反向写回待确认。

### 5.3 告警管理

#### 功能描述

告警管理用于汇总设备数据中的报警信息，内部人员可查看和处理，客户移动端不展示告警详情。

#### 功能清单

| 功能模块 | 功能描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| 告警列表 | 汇总所有平台的电池告警 | P0 | 可按告警时间、类型、等级筛选 |
| 告警详情 | 查看告警来源、BT码、客户、订单、设备数据 | P0 | 告警可追溯到具体电池 |
| 告警状态 | 支持待处理、处理中、已处理、误报 | P0 | 告警处理状态可更新 |
| 告警处理记录 | 记录处理人、时间、处理说明 | P0 | 告警处理过程可追踪 |
| 转报修 | 严重告警可转为售后报修单 | P1 | 转报修后保留告警来源 |

#### 关键规则

- 客户移动端不展示告警详情。
- 告警必须关联 BT码。
- 如果 BT码已绑定客户订单，告警应可显示客户和订单信息。

### 5.4 租赁订单管理

#### 功能描述

租赁订单以客户和电池为核心，由销售创建。创建订单时不绑定电池，电池在出库时绑定。

#### 订单主流程

```text
销售创建租赁订单
→ 填写客户、租期、价格、账单日等信息
→ 填写或上传合同信息
→ 发起销售租赁合同会签
→ 审批通过后订单生效
→ 订单进入待出库状态
→ 资产管理部门按订单出库并绑定 BT码
→ 客户确认收货
→ 系统按月生成账单
```

#### 功能清单

| 功能模块 | 功能描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| 订单创建 | 销售创建客户租赁订单 | P0 | 订单可保存并关联客户 |
| 合同信息 | 合同附属于订单，可上传附件或填写合同信息 | P0 | 订单详情可查看合同信息 |
| 合同会签 | 订单发起销售租赁合同会签 | P0 | 审批通过后订单进入待出库 |
| 订单账单日 | 每个订单可单独配置账单日 | P0 | 系统按订单账单日生成账单 |
| 订单电池列表 | 展示订单已绑定的 BT码电池 | P0 | 出库绑定后订单展示电池 |
| 部分出库 | 一个订单允许部分出库 | P0 | 订单可存在未全部交付状态 |
| 分批出库 | 一个订单允许多次出库 | P0 | 订单下可查看多个出库单 |
| 订单财务信息 | 查看应收、已收、欠款、逾期 | P0 | 财务和有权限客户可查看账单 |
| 订单售后记录 | 查看订单下电池报修记录 | P1 | 报修可关联到订单 |

#### 订单状态

```text
草稿
审批中
待出库
部分出库
待客户收货
租赁中
已完成
已取消
```

#### 关键规则

- 一个订单只属于一个客户。
- 一个订单可以绑定多组电池。
- 一个订单允许多次出库。
- 一个 BT码同一时间只能绑定一个有效租赁订单。
- 合同为订单附属信息，不作为主流程入口。

### 5.5 固定资产管理

#### 功能描述

固定资产管理负责电池台账、入库、出库、维修、报废、买断等资产流转。

#### 默认资产状态

```text
在库
维修
租赁
报废
买断
```

状态支持后台配置，但关键状态流转应由业务动作触发。

#### 功能清单

| 功能模块 | 功能描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| 电池台账 | 按 BT码管理电池基础信息 | P0 | BT码唯一且可查询 |
| 入库管理 | 采购、维修完成等场景入库 | P0 | 入库后电池状态为在库 |
| 出库管理 | 基于订单创建出库单并绑定 BT码 | P0 | 出库后电池状态为租赁 |
| 分批出库 | 同一订单可多次出库 | P0 | 订单下可查看多个出库单 |
| 收货确认 | 客户按出库单确认收货 | P0 | 客户确认后出库单变为已收货 |
| 自动收货 | 出库 7 天后客户未确认，系统自动收货 | P0 | 超时自动更新收货状态 |
| 调拨管理 | 支持内部资产调拨 | P1 | 调拨记录可追踪 |
| 报废管理 | 记录报废审批和结果 | P1 | 报废后状态不可出库 |
| 买断管理 | 记录客户买断流程和结果 | P1 | 买断后状态不可再租赁 |
| 资产履历 | 记录每组电池所有流转历史 | P0 | 可查看 BT码完整流转记录 |

#### 出库规则

- 出库必须关联有效租赁订单。
- 出库由资产管理部门操作。
- 出库时绑定 BT码。
- 出库前电池必须为“在库”状态。
- 出库成功后电池绑定客户和订单。
- 出库成功后电池资产状态变为“租赁”。
- 一个订单允许部分出库和分批出库。
- 客户按出库单确认收货。
- 出库后 7 天客户未确认，系统自动默认已收货。

#### 维修后规则

- 电池维修完成后必须重新入库。
- 重新给客户使用时，必须再次出库。
- 维修完成但未入库时，资产状态仍可保持“维修”，报修单状态体现“待入库”。

### 5.6 售后报修

#### 功能描述

售后报修支持客户移动端报修和内部后台报修。客户通过扫描电池二维码或输入 BT码发起报修，技术人员也可在后台对仓库或内部检测异常电池发起报修。

#### 客户报修流程

```text
客户登录移动端网页
→ 扫描电池二维码或输入 BT码
→ 系统校验 BT码是否属于当前客户有效订单
→ 不属于则提示“该电池不属于您的订单”
→ 属于则填写故障描述、上传图片、填写地址
→ 客户点击提交
→ 后台生成报修单
```

#### 功能清单

| 功能模块 | 功能描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| 客户扫码报修 | 扫描电池二维码识别 BT码 | P0 | 扫码后可识别 BT码 |
| 输入 BT码报修 | 客户手动输入 BT码 | P0 | 输入有效 BT码后可继续报修 |
| 客户电池校验 | 校验 BT码是否属于当前客户订单 | P0 | 非本客户电池不可提交报修 |
| 报修资料 | 填写故障描述、上传图片、填写地址 | P0 | 缺少必填项不可提交 |
| 内部报修 | 技术或资产人员可在后台发起报修 | P0 | 内部电池也可进入维修流程 |
| 报修受理 | 售后或技术人员受理报修 | P0 | 报修状态可更新 |
| 维修记录 | 记录处理过程、结果、费用、责任判断 | P1 | 维修过程可追踪 |
| 报修进度 | 客户查看报修状态 | P0 | 客户可查看自己的报修进度 |

#### 报修状态

```text
待受理
处理中
维修中
待入库
已完成
已关闭
```

#### 关键规则

- 客户报修必须上传图片和填写地址。
- 客户只能报修自己有效订单下的电池。
- 内部技术人员可对非客户订单电池发起报修。
- 维修完成后需走入库，再根据业务需要重新出库。

### 5.7 财务管理

#### 功能描述

财务管理负责租赁账单、回款、欠款、逾期、押金、采购付款等。当前核心为租赁订单月度账单和客户欠款管理。

#### 账单规则

- 租金按月生成。
- 租金按整月计算，不做按天折算。
- 每个订单可单独配置账单日。
- 账单生成后进入待确认状态。
- 财务确认后账单对客户生效。
- 财务可手动调整账单金额，但必须填写调整原因。
- 账单生成后 7 天未结清，系统自动标记逾期。
- 客户管理员可查看每期账单明细和欠款总额。
- 客户员工不可查看账单金额和欠款金额。

#### 功能清单

| 功能模块 | 功能描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| 自动生成账单 | 按订单账单日生成月度账单 | P0 | 到账单日自动生成应收记录 |
| 财务确认账单 | 财务确认系统生成的账单 | P0 | 确认后客户管理员可查看 |
| 手动调整账单 | 财务可调整金额并填写原因 | P0 | 调整记录可追溯 |
| 回款登记 | 财务登记客户付款 | P0 | 登记后更新已收和欠款 |
| 欠款统计 | 按客户、订单统计欠款 | P0 | 可查看欠款总额 |
| 逾期标记 | 账单生成 7 天后未结清自动逾期 | P0 | 逾期状态自动更新 |
| 账单明细 | 查看每期应收、已收、欠款、状态 | P0 | 财务可查看完整明细 |

#### 账单状态

```text
待确认
待付款
部分付款
已结清
已逾期
已调整
已作废
```

### 5.8 OA业务审批

#### 功能描述

OA 一期只做业务审批，不做人事行政全套。审批流程需要后台可配置，并支持审批通过后联动业务状态。

#### 一期审批范围

| 审批类型 | 优先级 | 说明 |
|----------|--------|------|
| 样品电池送样审批 | P1 | 样品送出、押金、运费等 |
| 采购需求审批 | P0 | 采购申请和明细 |
| 采购合同审批 | P0 | 采购合同会签 |
| 付款申请 | P0 | 采购、费用、维修等付款 |
| 销售租赁合同会签 | P0 | 租赁订单合同审批 |
| 采购入库申请 | P1 | 采购到货入库审批 |
| 返厂维修申请 | P1 | 电池返厂维修 |
| 人为损坏赔偿申请 | P1 | 人为损坏费用确认 |

## 7. 用户流程

### 7.1 租赁履约主流程

```text
销售创建订单
→ 合同会签
→ 订单生效
→ 资产出库绑定 BT码
→ 客户确认收货
→ 电池进入租赁状态
→ 系统生成月度账单
→ 财务确认账单
→ 客户查看账单和欠款
→ 财务登记回款
```

### 7.2 报修维修主流程

```text
客户扫码或输入 BT码
→ 系统校验客户和电池关系
→ 提交报修
→ 售后受理
→ 技术处理
→ 需要维修时电池进入维修状态
→ 维修完成
→ 资产重新入库
→ 需要继续使用时再次出库
```

### 7.3 设备数据同步流程

```text
配置第三方平台接口
→ 系统按频率主动拉取数据
→ 字段映射
→ 数据清洗
→ 按 BT码更新设备运行数据
→ 生成或更新告警
→ 内部人员查看处理
```

## 9. 数据字典

### 9.1 电池资产状态

| 状态 | 说明 |
|------|------|
| 在库 | 电池在公司仓库，可出库 |
| 维修 | 电池处于维修或待入库维修完成状态 |
| 租赁 | 电池已出库并绑定客户订单 |
| 报废 | 电池已报废，不可出库 |
| 买断 | 客户已买断，不再参与租赁 |

### 9.2 订单状态

| 状态 | 说明 |
|------|------|
| 草稿 | 销售创建但未提交 |
| 审批中 | 合同会签流程中 |
| 待出库 | 合同会签通过，等待资产出库 |
| 部分出库 | 已完成部分电池出库 |
| 待客户收货 | 出库后等待客户确认 |
| 租赁中 | 客户已确认或系统自动确认收货 |
| 已完成 | 订单履约完成 |
| 已取消 | 订单取消 |

### 9.3 出库单状态

| 状态 | 说明 |
|------|------|
| 待收货 | 已出库，客户未确认 |
| 已收货 | 客户手动确认收货 |
| 自动收货 | 出库 7 天后系统自动确认 |

### 9.4 账单状态

| 状态 | 说明 |
|------|------|
| 待确认 | 系统生成，待财务确认 |
| 待付款 | 财务确认，客户未付款 |
| 部分付款 | 已部分回款 |
| 已结清 | 已全额回款 |
| 已逾期 | 生成 7 天后未结清 |
| 已调整 | 财务调整过金额 |
| 已作废 | 账单作废 |

### 9.5 报修状态

| 状态 | 说明 |
|------|------|
| 待受理 | 报修已提交，等待处理 |
| 处理中 | 售后或技术人员处理中 |
| 维修中 | 电池进入维修流程 |
| 待入库 | 维修完成，等待资产入库 |
| 已完成 | 报修处理完成 |
| 已关闭 | 报修关闭 |

## 10. 风险与依赖

### 10.1 已知风险

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 8个平台字段不一致 | 数据清洗和展示复杂 | 先拿接口字段做统一字段模型 |
| BT码重复或缺失 | 资产、订单、报修数据串联错误 | 同步和录入时做唯一校验 |
| 状态配置过于自由 | 报表和流程混乱 | 状态可配置，但关键流转受业务流程控制 |
| 财务手动调整账单 | 金额风险 | 必须记录调整原因和操作日志 |
| 客户多账号权限 | 金额泄露风险 | 客户管理员和客户员工权限分离 |
| 分批出库 | 订单状态复杂 | 订单状态需要汇总多个出库单 |
| 自动收货 | 客户争议 | 客户端明确提示 7 天自动收货 |

### 10.2 外部依赖

| 依赖项 | 用途 | 风险等级 |
|--------|------|----------|
| 8个第三方平台接口 | 设备数据、告警数据同步 | 高 |
| 电池二维码 | 客户报修扫码读取 BT码 | 中 |
| 公司业务审批规则 | 配置初始审批流程 | 中 |
| 客户合同和租金规则 | 账单生成 | 高 |

## 12. 验收标准

- [ ] 系统支持内部员工账号和客户账号登录。
- [ ] 一个客户公司可创建多个客户账号。
- [ ] 客户管理员可查看金额，客户员工不可查看金额。
- [ ] 系统可配置第三方平台接口并主动拉取数据。
- [ ] 电池数据可按 BT码统一展示。
- [ ] BT码具备唯一性校验。
- [ ] 告警可关联 BT码、客户、订单。
- [ ] 客户端不展示告警详情。
- [ ] 销售可创建租赁订单并发起合同会签。
- [ ] 合同会签通过后订单进入待出库。
- [ ] 出库时可绑定 BT码。
- [ ] 非在库电池不可出库。
- [ ] 一个订单支持部分出库和分批出库。
- [ ] 客户可按出库单确认收货。
- [ ] 出库 7 天后未确认可自动收货。
- [ ] 系统按订单账单日生成月度账单。
- [ ] 租金按整月计算。
- [ ] 财务可确认和调整账单，调整必须留痕。
- [ ] 账单生成 7 天后未结清自动逾期。
- [ ] 客户可扫码或输入 BT码报修。
- [ ] 客户不能报修不属于自己订单的电池。
- [ ] 客户报修必须上传图片和填写地址。
- [ ] 内部技术人员可发起内部报修。
- [ ] 维修完成后必须重新入库，再出库给客户。
- [ ] OA审批流程支持后台配置。
- [ ] 审批通过后可联动业务状态。
